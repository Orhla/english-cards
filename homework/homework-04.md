## Отступление: почему мы не разбили функцию на слои

У меня есть "правильное архитектурное решение": разбить функцию на три части — отдельно HTTP-клиент к Яндексу, отдельно логика кэширования на диск, отдельно сервис который их объединяет.
Это _правильное_ разделение. Каждый слой отвечает за одно: клиент знает как говорить с Яндексом, кэш знает как работать с файлами (в проф проектах это может быть не "жесткий диск" а какое-то облачное хранилище), сервис знает когда и что вызывать.
Такая структура позволяет тестировать каждый кусок изолированно: в юнит-тестах сервиса мокаешь клиент, но файлы пишешь настоящие. Это не абстракция ради абстракции — это осознанный выбор в пользу тестируемости.
Но мы этого не сделали. Потому что наша функция — 40 строк. Три файла, интерфейсы и инъекции для 40 строк — это больше шума, чем пользы. Сам паттерн утонул бы в структуре.

Это и есть самая сложная задача в программировании — не написать правильный код, а найти **баланс**. Когда абстракция помогает, а когда мешает. Когда структура нужна, а когда это преждевременная сложность.
Однозначного ответа нет — он зависит от размера проекта, команды, и того, как будет жить код дальше. Опыт — это в том числе умение чувствовать эту границу.

## Setup

Создай новый файл: `src/lib/yandex-tts-service.test.ts` (перепиши старый).

Объяви переменную `let tmpDir: string` на уровне `describe` — вне любых хуков. Тесты будут читать её, а `beforeEach` будет присваивать перед каждым тестом.

**`beforeEach`** — создай уникальную временную папку и запиши путь в `tmpDir`. Используй `mkdtemp` из `fs/promises` — она создаёт папку с уникальным суффиксом, чтобы тесты не конфликтовали между собой.
Базовый путь возьми из `tmpdir()` (`import { tmpdir } from 'os'`). Замокай глобальный `fetch` через `vi.spyOn(globalThis, 'fetch')`.

**`afterEach`** — удали временную папку со всем содержимым. В `fs/promises` есть `rm` с опцией `{ recursive: true }`. Сбрось все моки через `vi.restoreAllMocks()`.

Что мокаем: **только `fetch`**. Файловую систему не трогаем — пишем настоящие файлы в настоящую временную папку.

## Тесты

### Тест 1 — ранняя ошибка если ключ не передан

Вызови функцию с нормальной карточкой, но без `yandexApiKey` (или с пустой строкой). Проверь что функция бросает ошибку с понятным текстом. Проверь что `fetch` не был вызван ни разу.

### Тест 2 — пропускает генерацию если файл уже есть

Перед вызовом функции создай файл `hello.ogg` в твоей временной папке (обычный `writeFile` с любым содержимым). Вызови функцию с `audioDir = tmpDir`. Проверь что `fetch` не был вызван.

### Тест 3 — Яндекс вернул ошибку

Замокай `fetch` чтобы возвращал `{ ok: false, status: 500, text: async () => "Internal Server Error" }`. Проверь что функция бросает ошибку, в тексте которой есть `"Yandex API Error (500)"`.

### Тест 4 — успешный кейс: файл появился на диске

Это самый сложный тест. Нам нужно замокать `fetch` так, чтобы `response.body.pipeTo()` не просто резолвился — а реально записал байты в файл. Иначе файл не появится на диске и проверить нечего.

**Подсказка:** `body` в настоящем `Response` — это `ReadableStream`. Его можно создать вручную:

```ts
const stream = new ReadableStream({
    start(controller) {
        controller.enqueue(new TextEncoder().encode("fake audio data"));
        controller.close();
    }
});
```

Замокай `fetch` чтобы возвращал `{ ok: true, body: stream }`. Теперь `pipeTo` внутри функции получит настоящий стрим и реально запишет байты.

После вызова функции проверь два условия: функция завершилась без ошибки, и файл `hello.ogg` существует в твоей временной папке (используй `access` из `fs/promises` — если файл есть, ошибки не будет).

### Тест 5 — ошибка при записи: битый файл удаляется

Замокай `fetch` чтобы возвращал `{ ok: true, body: { pipeTo: vi.fn().mockRejectedValue(new Error("disk full")) } }`. Проверь что функция бросает ошибку. Проверь что файл `hello.ogg` в временной папке **не существует** — то есть функция его почистила за собой.

Для проверки отсутствия файла: вызови `access` и убедись что она бросает ошибку с кодом `ENOENT`.
